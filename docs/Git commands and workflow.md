# Git 日常开发核心命令与实用工作流

本文档旨在提供一份在日常开发中高频使用的 Git 命令清单，并将其融入一个完整、高效、安全的工作流中。这套流程以“功能分支”模型为核心，适合个人开发者及团队协作。

---

## 核心工作流：从新功能到合并的全过程

这部分将模拟一个完整的开发任务，从接到需求到功能上线的全过程，让你理解命令在何时、何地、为何被使用。

### 准备阶段：同步最新代码，开启新任务

在开始任何新工作之前，永远要确保你的本地代码是基于远程仓库的最新版本，这可以最大程度地避免未来的合并冲突。

#### 第一步：切换到主分支
主分支（通常是 `main` 或 `master`）是我们所有工作的基线，是稳定代码的所在地。
```bash
# 切换到 main 分支
git checkout main
```

#### 第二步：拉取最新代码
从远程仓库（如 GitHub）拉取最新的提交，确保你的本地 `main` 分支与远程保持同步。
```bash
# git pull 是 git fetch 和 git merge 的组合
git pull origin main
```

### 开发阶段：创建分支与编码提交

为了不污染主分支的稳定性，所有新功能的开发、Bug 的修复都应该在一个独立的分支上进行。

#### 第一步：创建并切换到功能分支
为你的新任务创建一个描述性的分支名。例如，开发一个登录功能，可以命名为 `feature/login`。
```bash
# -b 参数会创建新分支，并立即切换过去
git checkout -b feature/login
```
现在，你就可以在这个“平行宇宙”里安心地编码了，任何操作都不会影响到 `main` 分支。

#### 第二步：编写代码与频繁提交
这是工作流的核心循环。你应该将一个大任务拆解成多个小步骤，每完成一步就进行一次提交。

1.  **查看状态**：随时使用 `git status` 查看当前工作区的变动情况。这是 Git 中最重要的命令，没有之一。
    ```bash
    # 查看哪些文件被修改、哪些文件是新增的
    git status
    ```

2.  **添加到暂存区**：将你希望本次提交包含的文件修改添加到暂存区。
    ```bash
    # 添加单个文件
    git add path/to/your/file.js

    # 添加所有已修改和新增的文件（最常用）
    git add .
    ```

3.  **执行提交**：将暂存区的内容创建一个新的版本快照，并附上清晰的描述信息。
    ```bash
    git commit -m "Feat: Add user login form structure"
    ```
    **小贴士：编写高质量的 Commit Message**
    一个好的 Commit Message 是最好的开发日志。推荐遵循约定式提交规范：`feat:` (新功能), `fix:` (修复 Bug), `docs:` (文档变更), `refactor:` (重构) 等。

### 合并阶段：将成果汇入主线

当功能分支上的任务开发完毕并通过测试后，就需要将其合并回主分支。

#### 第一步：切换回主分支并再次更新
```bash
git checkout main
git pull origin main
```
这一步是为了确保 `main` 分支在你开发期间没有被其他人更新，避免潜在的合并问题。

#### 第二步：合并功能分支
```bash
git merge feature/login
```
这条命令会将 `feature/login` 分支上的所有提交历史应用到当前的 `main` 分支上。

### 收尾阶段：推送与清理

合并完成后，需要将本地的更新推送到远程仓库，并清理掉已经完成使命的功能分支。

#### 第一步：推送主分支
```bash
git push origin main
```

#### 第二步：删除已合并的分支
```bash
# -d 参数会安全地删除已经合并过的分支
git branch -d feature/login
```
如果需要强制删除一个未合并的分支，可以使用 `-D` 参数。

## 常用辅助与查询命令

这些命令在日常工作中用于检查状态、查看历史和处理一些意外情况。

### 查看提交历史：git log
`git log` 可以展示从近到远的提交历史。
```bash
# 查看详细历史
git log

# 以单行简洁模式显示
git log --oneline

# 显示分支图，更直观
git log --graph --oneline --all
```

### 比较文件差异：git diff
用于查看工作区文件与暂存区、或与最新提交版本之间的差异。
```bash
# 查看工作区与暂存区的差异（即尚未 add 的修改）
git diff

# 查看暂存区与最新提交的差异（即已经 add 但未 commit 的修改）
git diff --staged
```

### 版本回退与撤销操作
**警告：撤销操作可能会导致工作内容丢失，请谨慎使用。**

-   **撤销暂存**：将已 `add` 到暂存区的文件移出，但保留工作区的修改。
    ```bash
    git reset HEAD <file>
    ```

-   **撤销工作区的修改**：放弃某个文件的所有本地修改，恢复到和最新提交版本一致的状态。
    ```bash
    # 这个操作会永久丢失你对该文件的修改，无法恢复！
    git checkout -- <file>
    ```

### 临时保存工作：git stash
当你正在一个分支上工作，但需要紧急切换到另一分支处理问题时，可以使用 `git stash` 临时保存当前工作区和暂存区的状态。
```bash
# 保存当前工作状态
git stash

# ...切换到其他分支处理紧急任务...

# 任务处理完后，切回原分支
git checkout feature/login

# 恢复之前保存的工作状态
git stash pop
```
